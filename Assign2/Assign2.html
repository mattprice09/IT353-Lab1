Q: How is the GET method different from POST when it comes to HTML form submission?

A: The GET method is different than the POST method with HTML form submission in that the GET method displays the user's form input in the address bar while redirecting, whereas the POST method sends the user's form input in the HTTP message body. Also, when the user hits the "Back" button, nothing happens with GET, whereas if a POST method was used, the data will be re-submitted. If you're worried about security and/or privacy, POST is the way to go.
Q: Identify several factors/reasons (min of 3 each) for choosing one method (GET/POST) over the other.

A:
GET POST
The "back" button doesn't require the user to re-submit the form/data. The user must re-submit forms/data because the data wasn't cached or kept in the history. This makes POST more secure than GET.
If the developer isn't concerned with security, the data is insensitive, or there aren't any important changes after the request, using GET makes navigating easier and more user-friendly than if the request were POST. If the data is sensitive, or the developer would like to provide more security/privacy, then POST should be used over GET. As stated in Part A, POST doesn't include the user's data as part of the URL. A good use case of POST would be when authenticating credentials. Using GET would defeat the purpose of having password characters hidden while typing. Also, the user's input data isn't cached or stored in browser history, as stated above.
A GET request is really most useful when you need to retrieve data (particularly, public data) and there aren't sensitive messages being sent. POST has no restrictions on the data's length, whereas, because the GET method appends the data to the URL, there is a max length of 2048 characters. This means that POST is a better choice if you have a lot of user input.
Q: What does it mean when one gets an HTTP status code in the range of 400-499? What about in the range of 500-599?

A: An HTTP status code in the range of 400-499 means that there was a client error. An HTTP status code in the 500s means that there was a server error.
Q: What does HTTP status code 403 mean? What about 404? What about 500?

A:
403 Forbidden - This essentially happens when the user doesn't have permissions for the requested resource.
404 Not Found - This indicates that the requested resource was not found.
500 Internal Server Error - This HTTP status code states that there was some error, but it was unable to provide more information. As a developer, this error can be a nightmare if you aren't familiar with the entire framework of the web app.
